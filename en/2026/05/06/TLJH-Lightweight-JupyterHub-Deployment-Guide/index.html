What is TLJH and Why Choose It

1.1 One Server = The Entire Team’s Development Environment

The Littlest JupyterHub (abbreviated TLJH) is the official lightweight multi-user computing platform from the Jupyter project. Its core selling point is dead simple:

Use a single ordinary Linux server to provide the entire team with a browser-ready Jupyter environment.

Once deployed, all you need to do is share the server IP with your colleagues. They open a browser, enter their credentials, and immediately get an independent Jupyter workspace, with pandas, scikit-learn, PyTorch, and other libraries you pre-installed ready to go out of the box.

1.2 The Best Solution When Resources Are Tight: TLJH vs Vanilla JupyterHub

JupyterHub itself is already a mature multi-user Jupyter platform, but it’s designed by default for medium-to-large organizations: it requires a separate database, a scalable proxy layer, and complex authentication configuration. TLJH (The Littlest JupyterHub) is the official streamlined distribution designed for the single-server scenario, and the core difference lies in its trade-off design when resources are tight.

Comparison TLJH Vanilla JupyterHub (Manual Deployment)
Number of servers 1 is enough At least 2 (Hub + separate proxy/database)
Minimum spec 2-core 4GB cloud server 4-core 8GB minimum, separate database adds overhead
Deployment method One command, 5 minutes Manual configuration of Hub, Proxy, database, Systemd
User isolation System-level users + shared Conda environment Container/Docker isolation, independent image per user
Resource usage Only 128MB base overhead Single user container takes hundreds of MB, Hub itself extra
Scale limit 1, 00 users Unlimited, but each additional node adds cost

The key trade-off is the level of user isolation:

  • Vanilla JupyterHub recommends Docker isolation by default, with each user in an independent container. This is indeed more secure and more “cloud-native,”but on a 2-core 4GB server, spinning up 5 user instances can consume all available memory.
  • TLJH’s choice is system-level users + shared Python environment. All users run on the same operating system, sharing the Conda environment under /opt/tljh/user. No container overhead, no image layering, 128MB of base memory sustains the Hub itself, and the remaining resources are entirely available for users’ actual computation.

Bottom line: If you only have one server and a team of fewer than 100 people, TLJH is the only solution that can actually run under resource constraints. It sacrifices container-level strong isolation in exchange for the ability to serve more users on limited hardware, for a trusted internal team environment, this is a perfectly reasonable trade-off.

1.3 TLJH Architecture Overview

1.4 These Scenarios Are a Perfect Fit for TLJH

  • Teaching/Training: Instructor pre-installs the course environment, students log in and start using it, say goodbye to “environment setup as the first lesson”
  • Data analysis teams: Unified Python versions and dependencies, eliminating “it works on my machine”once and for all
  • Lightweight GPU sharing: Single-GPU server with multi-user time-sharing, more VRAM-efficient than Docker solutions
  • Remote development environment: Replace local Anaconda, access from any browser anywhere, seamless migration across machines

Connecting to the Server

This guide assumes you’re using a cloud server (Tencent Cloud, Alibaba Cloud, Huawei Cloud, etc.).

  1. Get server information: Public IP, admin account (usually root), password
  2. Open a terminal:
  • Windows: Press Win + R, type cmd, or open PowerShell
  • macOS/Linux: Open Terminal
  1. Establish SSH connection: ssh root@
  2. Authentication:
  • On first connection, if you see a security confirmation prompt, type yes and press Enter
  • Enter your password (Linux doesn’t echo password input, just type it and press Enter)

Deploying TLJH

3.1 System Requirements

  • OS: Ubuntu 22.04+ or Debian 11+
  • Architecture: amd64 or arm64
  • Minimum spec: 2 CPU cores / 4GB RAM / 20GB disk
  • Recommended spec: 4 CPU cores / 8GB RAM / 50GB disk (team of up to 10)

3.2 The Core Question: Slow Installation on China Cloud Servers, What to Do?

This is the most critical part of this guide.

TLJH’s official install script needs to pull code from GitHub and install dependencies from PyPI. But cloud servers in China have extremely slow access to GitHub and the official PyPI, sometimes timing out entirely. You don’t need to configure a proxy for the entire server, just specify mirror sources temporarily during installation.

  • TLJH_GIT_MIRROR: Proxies GitHub raw content and git repository requests. ghproxy.net is a commonly used GitHub acceleration service in China; you can also substitute https://mirror.ghproxy.com/ or other trusted proxies
  • TLJH_PYPI_MIRROR: Proxies Python package downloads. pypi.tuna.tsinghua.edu.cn is the Tsinghua University TUNA mirror, stable and well-synced

3.3 Deployment Flowchart

3.4 Wait for Installation to Complete

The installation process typically takes 5, 5 minutes, and the terminal will continuously output logs. When you see Done! or return to the command prompt, deployment is complete.

During installation, TLJH automatically handles the following:

  • Installs Python 3, pip, git, and other base dependencies
  • Creates isolated Hub and user environments
  • Configures systemd services for auto-start on boot
  • Sets up Traefik reverse proxy, automatically handling HTTP/HTTPS

Global Python Environment Configuration

4.1 Why Global Configuration Is Needed

TLJH comes with a built-in Miniforge-based Conda environment under /opt/tljh/user, no need to install Anaconda/Miniforge yourself. All regular users share the Python libraries in this environment by default.

This means: the admin installs once, and everyone can use it.
