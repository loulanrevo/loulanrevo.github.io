JupyterHub Terminal + OpenCode：在浏览器里跑 AI 并行开发

本方案面向需要在低配置云服务器（如 2C4G 或共享实例）上进行 AI 辅助开发的用户。通过 JupyterHub Terminal + tmux + OpenCode 的组合，你可以在单台资源紧张的服务器上并行推进多个任务，同时保证任务不会因网络波动或浏览器关闭而中断。

为什么需要这套方案

现实痛点：资源紧张下的开发困境

在实际的科研和小型项目场景中，我们往往面临以下约束：

  • 预算有限：只能负担低配置云服务器（2核4G 甚至更低），无法本地运行大型 IDE
  • 网络不稳定：校园网或公共 WiFi 时常波动，SSH 连接容易断开
  • 任务需要并行：比如一边让 AI 探索特征工程思路，一边让 AI 实现已确定的特征
  • 后台任务不能断：模型训练、数据处理等长时任务，一旦中断就要重来

传统的本地 IDE（VS Code、PyCharm）在这种环境下几乎无法使用——它们本身就要占用大量内存，而且一旦网络断开，所有运行中的进程都会随 SSH 会话一起被终止。

这套方案的优势

痛点 传统方案 本方案
内存占用 VS Code Server 占用 1G+ Terminal + tmux 几乎零额外开销
网络中断 所有进程终止 tmux 会话保持，任务后台继续
并行开发 需要开多个 SSH 窗口管理混乱 两个 tmux 会话，逻辑清晰
AI 辅助 依赖本地 Copilot，配置复杂 OpenCode 直接读取自定义 skills
技能同步 手动复制粘贴 prompt 直接上传 skills 文件夹覆盖

核心思路：把 JupyterHub 的 Terminal 当作轻量级入口，tmux 作为会话守护者，OpenCode 作为 AI 助手。三者组合，在资源极度紧张的环境下也能实现”探索”与”实现”并行推进。

0. 环境与整体思路

  • 环境：Ubuntu 云服务器（已部署 JupyterHub）+ Terminal + tmux + OpenCode
  • 任务拆分：一个 Terminal 负责”特征工程探索”，另一个负责”特征工程实现”，并行运行
  • 后台保护：每个任务被放入独立的 tmux 会话。关闭浏览器或网络中断后，任务仍会在服务器后台继续执行

如果不使用 tmux 而直接在 Terminal 中运行任务，一旦关闭标签页或网络波动，任务大概率会被终止。除极短测试外，全程使用 tmux 是稳妥的做法。

1. 前置操作（仅初次需要）

1.1 开启 8 GB Swap（需 sudo）

对于 4G 内存的服务器，Swap 不是可选项，而是必需品。OpenCode 运行时会占用一定内存，加上 Python 数据处理，物理内存很快耗尽。没有 Swap，OOM Killer 会直接终止你的进程。

完成后执行 free -h，确认 Swap 一行显示 8.0Gi。若当前账号无 sudo 权限，可将上述命令转发至服务器管理员执行。

2. 上传本地 skills 文件夹并覆盖 OpenCode 全局路径

OpenCode 从 ~/.config/opencode/skills/<技能名>/SKILL.md 读取知识库。本步骤将你本地整份 skills 文件夹上传至服务器，并以 cp -r 完整覆盖到对应技能目录。

2.1 通过 JupyterHub 直接上传文件夹

在 JupyterHub 的文件管理界面（地址栏通常以 /tree 结尾），将你电脑上的 skills 文件夹整体拖拽到文件列表区域，即可上传。上传完成后，该文件夹会出现在服务器主目录下（例如 /home/你的用户名/skills）。

5. 常见问题

现象 可能原因 处理方式
opencode 提示 command not found 安装未完成或 PATH 未刷新 重新打开 Terminal 再试
tmux attach 无法进入会话 会话已丢失（如服务器重启） 执行 tmux ls 查看；若列表为空，需重建会话
任务运行缓慢或中断 内存紧张 在 btop 中确认 Swap 已启用；若 Swap 也耗尽，可考虑降低并行度（参数详见 OpenCode 文档）
上传的文件夹名称非 skills 上传时保留了原名 将后续命令中的 ~/skills 替换为实际文件夹名

6. 补充说明

  • 两个任务建议使用不同的输出路径或文件名，避免互相覆盖。
  • Swap 文件无需删除，可保留供后续使用。
  • 即使关闭浏览器，只要 tmux 会话存在，任务就不会中断。

7. 方案总结：为什么这样好

在资源紧张的场景下，这套方案的价值不仅在于”能跑起来”，更在于以最小的系统开销获得最大的开发效率：

零 GUI 开销：JupyterHub Terminal 是纯文本界面，不占用显卡或大量内存；相比远程桌面或 VS Code Server，节省的资源可以直接分配给实际任务。

会话持久化：tmux 是终端复用器中的经典工具，它将终端会话与 SSH 连接解耦。网络闪断、浏览器崩溃、笔记本合盖——都不会影响服务器上正在运行的进程。对于需要数小时甚至数天的特征工程任务，这是必需的保障。

  • AI 技能可移植：通过 skills 文件夹，你可以把本地精心调优的 prompt 和知识库一键同步到服务器。OpenCode 会按照 SKILL.md 的指引工作，相当于给 AI 装上了”领域大脑”。
  • 并行不混乱：探索与实现是两个认知阶段，串行执行会浪费大量等待时间。用两个 tmux 会话并行推进，一边试错一边落地，整体效率接近翻倍——而在低配置机器上，这种时间节省尤为珍贵。
  • 可扩展性：这个模式不局限于特征工程。模型调参、数据清洗、代码重构——任何可以拆分为”探索+实现”或”任务 A+任务 B”的场景，都可以复用同一套工作流。

如果你也在用一台”刚好够用”的服务器做开发，不妨试试这套组合。它不会让你的机器变快，但会让你在有限的资源里，挤出更多实际的产出。
